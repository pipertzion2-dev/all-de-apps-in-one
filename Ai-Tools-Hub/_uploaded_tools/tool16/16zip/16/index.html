What Is Schema Tightness?

Schema tightness measures how precisely a JSON Schema or OpenAPI definition constrains its data. A tight schema leaves little room for ambiguity: fields are marked as required where applicable, types are specific (integer, boolean, enum) rather than generic strings, and nesting is kept shallow enough for validators — and LLM function-calling — to reason about reliably.

This JSON Schema linter calculates a tightness score using three signals:

  • Required field ratio — the percentage of properties marked required. The higher, the better.
  • Strict type ratio — properties using enums or non-string types score higher than bare "type": "string" declarations.
  • Nesting depth — deep structures lose points because they are harder to validate and more likely to contain implicit optional paths.

A score of 70 or above is a solid baseline for production APIs. Anything below 40 typically means your API will silently accept malformed payloads.

Why Tight Schemas Matter for APIs and AI

Loose schemas create three categories of problems in production.

For REST APIs, under-constrained schemas mean your server must handle garbage inputs — null where an integer was expected, missing required fields treated as optional, freeform strings where only three values are valid. Each of these becomes a production bug or a security surface.

For OpenAI function calling and LLM tool use, schema tightness directly affects model accuracy. Models make significantly fewer errors calling functions defined by tight schemas than loose ones. Every optional field is a decision point where the model can hallucinate a value or omit a critical one — tight schemas eliminate that ambiguity at the definition layer.

For teams, tight schemas double as living documentation. When a field is required with a specific type, there is nothing left to interpret. New engineers and external consumers understand the contract immediately, without reading prose documentation.
