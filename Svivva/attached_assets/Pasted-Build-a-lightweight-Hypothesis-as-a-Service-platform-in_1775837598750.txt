Build a lightweight “Hypothesis-as-a-Service” platform inside of the digital side of svivva that allows users to explore APIs and automatically generate, test, and validate hypotheses about relationships between data sources.

Core Concept:
The system acts as an AI researcher. Instead of requiring users to define logic, it autonomously:
- generates hypotheses
- runs experiments across APIs
- evaluates results
- produces validated insights

This is not an API marketplace or workflow tool. It is a discovery engine that turns APIs into knowledge.

-------------------------------------

CORE USER FLOW (EXTREMELY SIMPLE UI):

1. Home Screen
- Single input:
  “What do you want to discover?”
- Example prompts:
  - “What affects X?”
  - “Find hidden patterns in Y”
- Button: “Discover”

2. API Selection (Card UI)
- Show APIs as simple cards:
  - name
  - short description
  - example output
  - “+ Add”
- Allow selecting 2–4 APIs max

3. Discovery Mode (One Click)
- Button: “Run Discovery”
- Show live progress:
  - “Generating hypotheses…”
  - “Designing experiments…”
  - “Testing relationships…”
  - “Analyzing results…”

4. Results (Insight Cards)
Each result should display:
- 🧠 Hypothesis
- 🔬 Result (confirmed / rejected / unclear)
- 📊 Insight (plain English)
- 📈 Confidence score
- 🔗 APIs used

Actions:
- Remix
- Save
- Turn into API

-------------------------------------

CORE FEATURES:

1. API Registry
- Users can add APIs:
  - endpoint URL
  - input schema (JSON)
  - sample response
- Each API gets:
  - public page
  - test playground (no login required)

2. Hypothesis Generation Engine (KEY FEATURE)
- Given selected APIs, generate multiple hypotheses:
  - correlations (A increases when B increases)
  - inverse relationships
  - anomalies
  - conditional behaviors

- Use LLM prompting to create hypotheses in natural language

3. Experiment Engine
- For each hypothesis:
  - generate multiple input scenarios
  - call APIs with varied inputs
  - store outputs with timestamps

4. Insight Validation Engine
- Compare outputs across experiments
- Detect:
  - consistent patterns
  - statistical differences
  - contradictions
- Assign confidence score (simple heuristic or statistical method)

5. Insight Memory
- Store:
  - hypotheses
  - experiments
  - results
- Prevent duplicate testing
- Allow incremental learning

6. Remix System
- Users can:
  - edit hypothesis
  - rerun experiments
  - refine insights

7. Insight Feed
- Display discoveries as a feed
- Each item:
  - clickable
  - remixable
- Acts as discovery + distribution layer

-------------------------------------

TECHNICAL CONSTRAINTS (REPLIT-OPTIMIZED):

- Backend: Node.js (Express) or Python (Flask)
- Frontend: React or simple JS
- Database: SQLite (or JSON for MVP)
- API calls: async fetch/requests
- Background processing:
  - simple interval loops (no queues)

-------------------------------------

INSIGHT ENGINE (MVP LOGIC):

Start simple:

- Run APIs with varied inputs
- Compare outputs using:
  - difference thresholds
  - trend detection
  - basic correlation logic

Optional:
- Use LLM to summarize findings into human-readable insights

-------------------------------------

UX PRINCIPLES:

- No login required for exploration
- No technical setup
- No workflow builders
- Minimal screens (input → select → results)
- Time to first insight < 10 seconds

-------------------------------------

AVOID:

- Over-engineering
- Complex ML pipelines
- Heavy infrastructure
- Manual configuration-heavy UI

-------------------------------------

GOAL:

Create a system where users can ask questions about relationships between data sources, and the platform autonomously generates and tests hypotheses to produce new knowledge—making discovery as simple as search.
